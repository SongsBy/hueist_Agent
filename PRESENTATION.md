# HUEIST 최종 발표 대본 (5분)

> **녹화 방법**: 화면 공유로 앱(`npm run dev` → localhost:3000)을 띄워두고, 아래 대본을 읽으면서
> [시연] 블록의 동작을 직접 클릭해 보여주면 됩니다. 각 섹션 옆 `[mm:ss]`는 누적 시간 가이드입니다.
> 평가요소(**구동 / 설계-데이터플로우·논리설계**)에 맞춰 시연과 설계 설명에 시간을 몰아두었습니다.

---

## 0. 오프닝 — 문제 정의  `[00:00 ~ 00:35]`

> 안녕하세요. 디자인 비전공 1인 개발자를 위한 디자인 톤 생성 도구, **HUEIST**를 발표하겠습니다.
>
> AI가 코드는 다 짜주는 시대지만, 막상 "내 앱에 무슨 색을 써야 하지?"는 여전히 개발자의 마지막 벽입니다.
> 기존 도구(Figma Make, Google Stitch)는 `modern, clean...` 처럼 **'표현할 줄 아는' 디자이너**를 전제로 합니다.
> HUEIST는 정반대입니다. **색을 고르게 하지 않고, 분위기를 고르게 합니다.**
> HEX 코드를 묻지 않고, AI가 3장의 레퍼런스 카드로 좁혀주면 사용자는 마음에 드는 한 장만 고릅니다.

---

## 1. 솔루션 한 줄 + 핵심 차별점  `[00:35 ~ 01:05]`

> 핵심 차별점은 **레퍼런스 카드**입니다.
> - 기존: 프롬프트로 묘사 → 생성마다 색이 달라짐
> - HUEIST: **시각적 카드 선택 → 선택한 톤으로 앱 전체가 고정**
>
> 그리고 AI가 카드 3장을 만들 때 **서로 완전히 다른 배색 전략**을 강제합니다.
> 같은 입력이라도 ① 브랜드 강조형 ② 미니멀 에디토리얼 ③ 보색 파격형으로 갈라지죠.
> 이게 "색을 골라주는 도구"가 아니라 "선택지를 설계해주는 도구"인 이유입니다.

---

## 2. 구동 시연 (Live Demo)  `[01:05 ~ 02:35]`

> 실제로 동작을 보여드리겠습니다.

**[시연 1] 입력** — 홈(`/`)에서 **시작하기** → `/create`
> "어떤 앱인가요?"에 한 줄 입력 (예: *"동네 책방을 위한 중고책 거래 앱"*).
> 아래에서 베이스 레이아웃 템플릿(커머스/피드 등)도 하나 고를 수 있습니다.

**[시연 2] AI 톤 추천** — 제출 → `/api/recommend` 호출 → `/recommend`
> 잠시 뒤 **3장의 카드**가 뜹니다. 보시면 카드마다 컬러·폰트·무드가 확연히 다릅니다.
> 1번은 브랜드 컬러를 쨍하게, 2번은 거의 무채색 + 은은한 틴트, 3번은 보색 대비.
> **같은 입력인데 배색 구조 자체가 다르다** — 이게 아까 말한 전략 강제의 결과입니다.

**[시연 3] 선택 → 실제 화면 생성** — 카드 선택 → 하단 CTA → `/mobile-playground`
> 마음에 드는 카드를 고르면, 그 톤으로 **실제 모바일 앱 화면을 즉시 생성**합니다.
> 단순 팔레트가 아니라, 그 톤이 적용된 진짜 UI를 react-live로 렌더해서 보여줍니다.
> (채팅으로 "버튼을 더 크게" 같은 수정도 가능합니다.)

**[시연 4] 디자인 시스템 출력** — `/result`
> 최종 결과 화면입니다. 컬러 팔레트(주요/보조/배경/강조/표면), 폰트 페어링, 모바일 목업,
> 그리고 **WCAG 명도 대비 경고**까지 — 바로 개발에 가져다 쓸 수 있는 디자인 시스템입니다.

## 3. 설계 ① — 데이터 플로우  `[02:35 ~ 03:50]`

> 이제 내부 설계를 설명드리겠습니다. 핵심은 **2-Phase 생성 아키텍처**입니다.

```
[사용자 설문/프롬프트]
        │  survey { category, mood, color_temp/bright/sat, free_text }
        ▼
┌──────────────────────── Phase 1 : Moodboard (빠른 응답) ────────────────────────┐
│  POST /api/recommend                                                            │
│   1) buildBrief()      설문 → 서술형 한국어 브리프로 재구성                         │
│   2) callGemini()      gemini-3-flash-preview (폴백: 2.5-flash), thinkingBudget 0 │
│   3) JSON 3 tones      COLOR STRATEGY MATRIX(A/B/C) 강제                          │
│   4) normalizeTone()   chroma-js로 color_scales 생성 + 폰트 URL (외부호출 X)       │
└─────────────────────────────────────────────────────────────────────────────────┘
        │  recommendedTones[3]  →  Zustand store (localStorage 영속화)
        ▼  (사용자가 1장 선택)
┌──────────────────────── Phase 2 : Deep Dive (선택 1장만 보강) ───────────────────┐
│  POST /api/enrich          Promise.all 로 무거운 외부 API 3개 병렬 호출            │
│   ├ enrichTypography()     Google Fonts (weight map, fallback stack)             │
│   ├ fetchIconsByStyle()    Iconify 아이콘 세트                                    │
│   └ fetchPexelsImages()    Pexels 무드 이미지 (24h 캐싱)                          │
│   →  "Master JSON" (Figma 핸드오프까지 가능한 완성형)                              │
└─────────────────────────────────────────────────────────────────────────────────┘
        │
        ▼
┌──────────────────────── 생성형 UI : 실제 화면 ──────────────────────────────────┐
│  POST /api/generate-ui                                                           │
│   1) loadBaseTemplate() + preprocessTemplate()  Tailwind→인라인 변환(결정론적)     │
│   2) callGemini()                               톤에 맞춰 재색칠                   │
│   3) sanitizeCode()                             펜스/import 제거                   │
│   4) validateGeneratedUi()  ◀── 린터 게이트키퍼 루프(최대 3회, 위반사유 피드백 재생성)│
│   →  JSX 문자열  →  react-live 렌더                                               │
└─────────────────────────────────────────────────────────────────────────────────┘
```

> 왜 두 단계로 나눴냐면 — 카드 3장은 **빠르게** 보여줘야 하므로 외부 API를 안 부르고,
> 사용자가 **고른 한 장만** 무거운 Google Fonts·Iconify·Pexels를 병렬로 보강합니다.
> 불필요한 외부 호출을 3분의 1로 줄이는 구조입니다.

## 4. 설계 ② — 논리적 설계 (3가지 핵심 결정)  `[03:50 ~ 04:40]`

> 단순히 "AI에 물어본다"가 아니라, **신뢰성을 위한 논리 장치 3개**를 넣었습니다.

**① COLOR STRATEGY MATRIX — 제품의 존재 이유를 코드로 강제**
> 프롬프트에서 3개 톤에 Monochromatic / Minimal-Tint / Complementary 전략을 1:1로 강제합니다.
> "같은 색을 3톤 Primary에 반복하면 응답 무효"라고 못 박아, **선택지가 진짜 갈리도록** 보장합니다.

**② 린터 게이트키퍼 루프 — LLM 출력의 결정론적 검증**
> 생성된 UI 코드를 그냥 믿지 않습니다. `validateGeneratedUi()`로 계약(인라인 스타일만,
> 토큰 색상만, 미정의 아이콘 금지, `render()` 필수)을 검사하고, **위반하면 그 사유를 다음 요청에
> 피드백해 재생성**합니다(단순 재시도가 아니라 교정 루프). 끝까지 색만 틀리면 `remapRogueColors`로
> 가장 가까운 토큰에 맞춰 **빈 화면 대신 보정본을 렌더**하는 graceful degradation까지 둡니다.

**③ 모델 폴백 체인 + 지수 백오프 — 가용성**
> Gemini가 503/429를 흘릴 때를 대비해 `callGemini`가 모델 체인(예: 3-flash → 2.5-flash)을
> 자동 폴백하고, 모델당 지수 백오프 재시도를 합니다. 한 모델 쿼터가 차도 플로우가 안 끊깁니다.

---

## 5. 설계 ③ — 상태관리 · 보안 + 마무리  `[04:40 ~ 05:00]`

> 상태는 **Zustand + persist(localStorage)**로 페이지 간 톤·설문을 공유하고 새로고침에도 복원합니다.
> 휘발성 UI 상태(isEnriching 등)는 `partialize`로 영속화에서 제외했습니다.
>
> 보안은 **모든 Gemini 호출을 서버 API Route로만** 경유시켜 API 키를 클라이언트에 노출하지 않습니다.
>
> 정리하면 HUEIST는 ― **"분위기를 고르면 AI가 좁혀주는"** 컨셉을,
> **2-Phase 데이터 플로우 + 전략 강제 + 결정론적 검증**이라는 설계로 신뢰성 있게 구현한 도구입니다.
> 이상 발표를 마치겠습니다. 감사합니다.

### 부록 — 발표 전 체크리스트
- [ ] `cd hueist && npm run dev` 로 서버 기동, `.env.local`에 `GEMINI_API_KEY` 등 설정 확인
- [ ] 녹화 전 미리 한 번 추천을 돌려 **Gemini 응답/네트워크 워밍업** (첫 호출 지연 대비)
- [ ] 데모용 입력 문구 1개 고정 (예: "동네 책방 중고책 거래 앱")
- [ ] 화면 확대(브라우저 110~125%)로 카드·팔레트가 잘 보이게
- [ ] 데이터 플로우 다이어그램(섹션 3)은 화면에 띄워두고 짚으며 설명
